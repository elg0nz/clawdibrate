## Problem

The implementer has no hard constraint on output size. Rewrites can grow the file even when the score improvement is marginal. The existing ROI check uses edit distance (lines changed) but not token delta.

## Deliverable

1. **Pass token context to implementer prompt.** Add to the implement_prompt:
   - Current section token count
   - Remaining token budget
   - Instruction: "Your output MUST NOT exceed {current_tokens} tokens. Prefer compression."

2. **Token gate in orchestrator.** After implementer returns, measure new section tokens:
   - If `new_tokens > old_tokens` AND score improvement < threshold → reject
   - If `new_tokens > old_tokens` AND score improvement >= threshold → accept with warning
   - If `new_tokens <= old_tokens` → accept (compression is always good)

3. **Updated ROI formula:**
   ```
   token_roi = score_improvement / max(token_delta, 1)
   ```
   Reject when `token_roi < 0.005` (configurable).

4. **Total budget enforcement.** After all section fixes, check total file tokens against `--token-budget`. If over, reject the lowest-ROI changes until under budget.
