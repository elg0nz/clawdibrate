## Problem

The `--synthesize-git-history` transcript has `repetition_score: 0.75` when a section is edited repeatedly across commits, but the bug-identifier gets 0 failures because there's no conversational context — only `edit` tool calls.

The insight: sections edited 3+ times in git history are structurally unstable. The rules aren't landing clearly enough to stay put across sessions.

## Deliverable

1. **`analyze_section_churn()`** in `git_history.py` — parse unified diffs for each commit, extract which `## Section` headings had lines changed, return `{section: churn_count}`.

2. **Inject `section_churn`** into the `session_start` event of the synthesized transcript.

3. **Inject synthetic `user_message` events** (one per high-churn section, threshold=3) with content like: *"The section 'X' was edited N times across git history. This suggests the rule is unstable or unclear."* — gives the bug-identifier conversational context.

4. **`high_churn_sections`** field in `session_end` for easy filtering.

## Implementation

`clawdibrate/git_history.py` — `_sections_changed_in_diff()`, `analyze_section_churn()`, and updated `synthesize_transcript_from_git()`.
