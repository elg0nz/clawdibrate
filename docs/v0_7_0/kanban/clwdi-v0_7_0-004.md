## Problem

The implementer can make large rewrites that produce small gains — a classic overfitting signal. Currently the only guard is "never full-rewrite sections >= 0.8" but there's no quantitative penalty for edit size vs improvement.

## Deliverable

Add a regularization term to the implementer scoring:

1. **Measure edit distance.** Compute character-level edit distance (or line diff count) between old and new section content.
2. **Compute improvement-to-change ratio.** `score_improvement / edit_distance`. Small targeted edits with big gains = good. Large rewrites with small gains = reject.
3. **Threshold.** If the ratio falls below a configurable minimum (default TBD from empirical runs), reject the change and log it as "low ROI rewrite."
4. **Persist.** Log edit distances alongside scores in reflections.jsonl for trend analysis.
