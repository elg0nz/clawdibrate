## Problem

If the judge sees full AGENTS.md, it scores based on stylistic consistency rather than functional correctness. Verbose AGENTS.md files get inflated scores. More fundamentally, AgentBench avoids LLM-as-judge entirely — all 8 environments use deterministic checking pipelines (bash scripts, SQL hash comparison, F1 against gold answers, game win rates). Our judge should maximize deterministic scoring and minimize LLM judgment.

## Design: Two-tier scoring

### Tier 1: Deterministic (no LLM call, computed by orchestrator)

These are the primary metrics. They're cheap, reproducible, and directly measure token waste — the core goal of clawdibrate.

| Metric | Formula | What it measures |
|---|---|---|
| **Token efficiency** | `tokens_used / tokens_in_ideal_path` | How much token waste AGENTS.md prevented (or failed to prevent). Ideal path = minimum tool calls for the task type |
| **Search waste ratio** | `search_calls_for_known_info / total_search_calls` | Fraction of searches that targeted info already in AGENTS.md. 0.0 = perfect AGENTS.md, 1.0 = completely useless |
| **Correction rate** | `user_corrections / user_messages` | How often the user had to redirect the agent. High = AGENTS.md gaps |
| **Repetition score** | `repeated_tool_patterns / total_tool_calls` | Per AgentBench J.2.4: >90% of Task Limit Exceeded failures show repetition. Binary flag if Rouge-L > 0.8 in sliding window |
| **Success rate (SR)** | binary: did the agent complete the task? | AgentBench's primary metric across all environments. Binary yes/no, not a gradient |

### Tier 2: LLM judge (only for what deterministic metrics can't capture)

The LLM judge is invoked ONLY for qualitative assessment that can't be computed:

- Was the circuitous path due to an AGENTS.md gap, or an inherent LLM limitation?
- Is the suggested fix actually actionable, or would it bloat AGENTS.md without benefit?
- Does the failure pattern repeat across multiple transcripts (frequency assessment)?

The judge receives ONLY:
- The identified failure and its evidence from the transcript
- The specific AGENTS.md section responsible (NOT the full file)
- The deterministic metrics for that failure (from Tier 1)
- The rubric criteria

It does NOT receive the full AGENTS.md. It outputs:

```json
{
  "is_agents_md_failure": true,
  "fixability": "high",
  "reasoning": "Ticket naming format is non-discoverable info, belongs in Boundaries",
  "weight": 0.8
}
```

### Weighted composite score

```
failure_score = (
    0.40 * token_efficiency +
    0.25 * (1 - search_waste_ratio) +
    0.15 * (1 - correction_rate) +
    0.10 * (1 - repetition_score) +
    0.10 * success_rate
)
```

Weights emphasize token efficiency as the primary optimization target. The composite produces a 0.0–1.0 score that is mostly deterministic, with LLM judgment used only for the fixability/actionability gate.

### Why this matters

AgentBench Table 4 shows the dominant failure mode is Task Limit Exceeded (24.9% commercial, 36.9% OSS) — agents burning tokens in loops. For clawdibrate, every token wasted because AGENTS.md didn't provide the right context is a measurable cost. The judge should optimize for token reduction first, correctness second (correctness without efficiency is just expensive correctness).
