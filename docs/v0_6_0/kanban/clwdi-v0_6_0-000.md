## Problem

The current loop uses synthetic tasks that don't test real AGENTS.md utility. Meanwhile, the **actual signal** is sitting right in front of us: real agent conversations.

Example from this session's transcript:

```
⏺ Let me check the existing version structure and ticket conventions first.
  Searched for 1 pattern, read 3 files (ctrl+o to expand)
⏺ Good, I have the conventions. Now let me create v0.6.0...
```

The agent searched for ticket naming conventions and version directory structure — information that AGENTS.md should have provided upfront. This is a real AGENTS.md failure: the agent wasted tool calls discovering what the system prompt should have told it.

The loop can't detect this because it uses fake tasks. Real transcripts are the ground truth.

## Architecture

Delete all existing code (`clawdibrate/`, `clawdibrate-loop.py`). Replace with:

```
clawdibrate/
  prompts/
    bug-identifier.md    # Reads transcript, identifies AGENTS.md failures
    judge.md             # Scores each identified failure with rubrics
    implementer.md       # Rewrites AGENTS.md sections to fix failures
  orchestrator.py        # Thin loop: record → identify → judge → implement → repeat
```

### 1. Transcript capture

Two options for the user:
- **Skill-based**: `/clawdbrt:record-start` and `/clawdbrt:record-stop` — wraps the conversation and saves it as a transcript file to `.clawdibrate/transcripts/`
- **One-line AGENTS.md**: Add a line to any repo's AGENTS.md that tells the agent to log tool calls and decisions to a structured transcript file

Transcripts capture: tool calls made, files searched, time spent searching vs. acting, whether the agent asked clarifying questions it shouldn't have needed to.

### 2. Bug identifier (`prompts/bug-identifier.md`)

Reads a transcript and identifies failures. A "failure" is any place where the agent:
- Searched for something AGENTS.md should have told it (like this session: searching for ticket naming format)
- Used the wrong tool/command when AGENTS.md specifies the right one
- Violated a boundary rule
- Asked the user something AGENTS.md already answers
- Took a circuitous path when AGENTS.md should have provided a direct one

Output: structured list of `{failure, responsible_section, evidence_from_transcript, suggested_fix}`

### 3. Judge (`prompts/judge.md`)

Scores each identified failure using the existing rubrics (0.0–1.0). Invoked via the user's CLI agent (same as bug-identifier). Scores based on:
- Severity: how much time/tokens wasted
- Frequency: does this pattern repeat across transcripts
- Fixability: can AGENTS.md actually prevent this, or is it an LLM limitation

### 4. Implementer (`prompts/implementer.md`)

Takes scored failures and rewrites the responsible AGENTS.md section. Uses the existing tuning rules (concise, exact CLI commands, file paths over prose, under 700 words). Invoked via the user's CLI agent.

### 5. Orchestrator (`orchestrator.py`)

Thin Python script that:
1. Reads transcripts from `.clawdibrate/transcripts/`
2. Calls bug-identifier via CLI agent with `bug-identifier.md` as system prompt
3. Calls judge via CLI agent with `judge.md` as system prompt
4. Calls implementer via CLI agent with `implementer.md` as system prompt
5. Writes new AGENTS.md, bumps version, commits

The orchestrator does NO prompt engineering — all intelligence is in the prompt files. All three meta-agents are invoked via the same CLI the user already has installed (`claude`, `codex`, `llm`, etc.) with the prompt file passed as system prompt.

## Installation on any repo

Two paths:
1. `npx skills add clawdibrate` → installs the recording skill, user runs `/clawdbrt:record-start` before working, `/clawdbrt:record-stop` after
2. Add one line to existing AGENTS.md: `> Clawdibrate: log all tool calls to .clawdibrate/transcripts/{session}.jsonl` — then run `python -m clawdibrate calibrate`

## Why this is ticket 000

Every other ticket in v0.6.0 (001–011) describes bugs in the current synthetic-task loop. This ticket replaces that loop entirely with transcript-based evaluation. The other tickets become sub-tasks of this architecture:
- 001 (system prompt injection) → still relevant for passing meta-prompts as system prompts via CLI
- 003 (self-judging) → mitigated by separating identifier/judge/implementer into distinct prompts with distinct roles
- 006 (regex corruption) → still relevant for `replace_section()`
- 008 (reflection persistence) → still relevant
- Others are obsoleted or subsumed by the new architecture
